## Problem Statement

Build the full Smox Lab platform as specified in `docs/SmoxLab-PRD.md` and `docs/SmoxLab-SAD.md` — an immersive retail platform for custom-printed apparel combining in-store kiosks, AR try-on, AI-assisted design generation, instant DTG printing, a Gaming Lounge queue display, a web marketplace with creator monetization, and an admin dashboard. Multilingual from day one (AR/FR/EN with RTL). Morocco market first (with Prayer Pause as a P0 feature), then France and UAE. **Single-brand, multi-location business model:** one company operating multiple company-owned Smox Lab locations sharing a unified customer pool and creator marketplace. The PRD's earlier "SaaS to other retailers" framing is rejected — see `docs/adr/0000-multi-tenancy.md`.

The build target for this design doc is **a locally-running full system on a single Mac via `docker compose up`**: 3 frontends + 8 NestJS microservices + Postgres + Redis + mocked printer + mocked AI providers. Hardware integration with real DTG printers comes after the local system is complete.

1. **Walk-In Customer** (primary, end-user) — age 16–35, wants to design and walk out wearing/carrying a custom item.
2. **Fashion Creator / Influencer** — wants to publish designs and earn commissions.
3. **Store Operator** — manages orders, printer queues, sales.
4. ~~**Brand / Retailer (SaaS Customer)** — licenses the platform.~~ **Rejected.** Per `docs/adr/0000-multi-tenancy.md`, Smox Lab is a single-brand multi-location business, not a SaaS platform. The PRD's "3 pilot stores within 12 months" refers to additional company-owned Smox Lab locations, not licensee retailers. This persona is removed.

For this build, the deployment target is the **family-connected print/retail venue in Morocco** (specific shop name not provided in this session). The walk-in customer at that single store is the first real user.

## Constraints

- **Solo frontend developer** for v1. No backend hire planned. User accepts they will be learning NestJS/microservices/Postgres/Redis/Socket.io/IoT integration as part of the build.
- **Timeline:** Not a constraint. 18–36+ months of solo work is accepted.
- **Hardware:** Deferred. First milestone is local-only with mocked printer and mocked AI providers.
- **Markets at launch:** Morocco, France, UAE. Multilingual (Arabic with RTL, French, English) is P0 from day one.
- **Performance budget:** Kiosk UI ≥ 60 fps, AR overlay ≥ 24 fps minimum (per PRD).
- **Reliability:** ≥ 99.5% uptime during store hours (per PRD).
- **AI generation latency:** ≤ 6 seconds (per PRD).
- **Accessibility:** WCAG 2.1 AA on the web platform.
- **Data:** AR photos auto-delete after 24h unless opted in; no PII in logs anywhere.
- **Prayer Pause:** P0 for Morocco market — must work end-to-end across all 3 frontends and the print queue.
- **Offline:** Kiosk must function in degraded mode if internet drops (no AI, no AR cloud sync, but core flow works).
- **Stack lock-in (per CLAUDE.md):** Next.js 15, React 19, TypeScript strict, Tailwind, Framer Motion, Zustand (no Redux), Three.js + R3F + drei, MediaPipe Holistic (TF.js fallback), Konva preferred over Fabric, Socket.io.

## Premises

Revised in D5 after user pushback. The agreed premises are:

1. **The full PRD will be built end-to-end.** Nothing cut. No deferred-as-cut. All 8 microservices, all 3 frontends, AR + AI + canvas + marketplace + creator + admin + prayer pause + i18n + offline.
2. **The family-connected print/retail venue in Morocco is the deployment target.** The deployment timeline is open; the local Docker Compose milestone is the gate to hardware planning.
3. **The 24-month timeline is accepted.** Morale management over multi-year solo work is treated as a real engineering constraint, not a soft concern.
4. **Customer interviews are not a pre-requisite to building.** User accepts the risk that design decisions made in the absence of customer observation will require rework when real customers eventually use the system.

## Approaches Considered

### Approach A: Vertical Slices (rejected)
Build one user flow end-to-end first (idle → product → simple design → confirm → ticket → mocked print → done), then add features as further slices.
- **Pros:** Working software in 3 months; integration bugs surface early; morale stays alive over multi-year build.
- **Cons:** Service boundaries crystallize gradually; some early architectural decisions need refactoring.
- **Rejected by user** in D6.

### Approach B: Contracts-First / Backend-Up (CHOSEN)
Lock all DTOs, OpenAPI specs, WebSocket event schemas, and DB models in `libs/types` and migrations first. Build all 8 NestJS microservices to spec with stub implementations. Bring up gateway + Postgres + Redis. THEN build frontends against stable real APIs. Hardware/AI providers remain mocked behind the same interfaces.
- **Pros:** Service boundaries stable from day one; less contract refactoring later; supports clean scaling to additional company-owned Smox Lab locations without per-tenant partitioning overhead.
- **Cons:** 5–8 months with no demoable product; backend designed in a vacuum frequently misses real frontend needs; highest burnout risk for a solo dev.

### Approach C: Foundation-First Layered (rejected)
Build out the shared layer deeply before touching apps (`libs/types`, `libs/ui` with full design system, `libs/i18n` with AR/FR/EN+RTL, auth lib, websocket-client lib). THEN stand up infra. THEN build apps and services in parallel.
- **Pros:** Best long-term architectural hygiene; every later feature inherits from a polished base.
- **Cons:** Slowest path to anything visible (6–9 months); foundations designed without a real consumer often get rebuilt.
- **Rejected by user** in D6.

## Recommended Approach

**Approach B: Contracts-First / Backend-Up.**

The detailed phase-by-phase build plan follows. Each phase has an explicit deliverable and exit criteria. Phases are sequenced so that no phase depends on a later one. Suggested elapsed times assume solo work with AI pair-programming assistance; real timelines will vary by 30–50%.

### Phase 0 — Repo Hygiene & Conventions (~1 week)

**Goal:** Lock the conventions everything else inherits.

**Deliverable:**
- `tsconfig` strict mode locked in `libs/typescript-config` (already scaffolded — extend it).
- ESLint + Prettier rules across all apps and libs (already scaffolded — populate it).
- Conventional Commits + commit lint hook.
- Pre-commit hook: typecheck + lint on staged files.
- npm workspaces verified (already using `package-lock.json`; staying on npm for v1 — no migration overhead, npm 9+ supports workspaces fine for this monorepo).
- Turborepo task pipeline declared: `build`, `dev`, `test`, `lint`, `typecheck` with proper dependency graph.
- `README.md` documents `npm run dev`, `npm run build`, `npm run test`.

**Exit criteria:**
- `npm run lint` and `npm run typecheck` pass cleanly across all apps and libs (even though most are empty).
- A test commit with bad TypeScript fails the pre-commit hook.
- `npm run dev` (or equivalent) boots all 3 apps as blank Next.js shells with no errors.

### Phase 1 — Shared Types & Contracts (~6–8 weeks)

**Goal:** The single source of truth for every cross-service shape. Expect 30–40% of these types to be refactored once Phase 9 (kiosk) consumes them. This is the inherent cost of Contracts-First and is acknowledged — Phase 1's job is to get the contracts close enough that Phases 4–7 can build against them, not perfect enough that Phase 9 changes nothing.

**Deliverable in `libs/types/`:**
- `product.ts` — Product, ProductVariant, Color, Size, FabricSwatch.
- `design.ts` — Design, DesignLayer, TextElement, ImageElement, ShapeElement, AIGenerationRequest, AIGenerationResult.
- `order.ts` — Order (status: `DRAFT | PENDING | QUEUED | PRINTING | READY | PAID | PICKED_UP | ERROR`), Ticket, OrderLineItem.
- `printer.ts` — PrinterJob, PrinterStatus (`ONLINE | PRINTING | ERROR | OFFLINE | MAINTENANCE`), PrinterCapability.
- `user.ts` — User, CustomerProfile, CreatorProfile, OperatorProfile, AuthToken.
- `creator.ts` — PublishedDesign, CreatorEarnings, CommissionRate, PayoutRequest.
- `kiosk.ts` — KioskSession, KioskState.
- `prayer.ts` — PrayerSchedule, PrayerName (`FAJR | DHUHR | ASR | MAGHRIB | ISHA`), PrayerPauseState.
- `events.ts` — All WebSocket event payload types (order status changes, printer state changes, prayer pause start/end, kiosk session start).
- `api/` — Per-service request/response DTOs for every REST endpoint (one file per service: `api/order.ts`, `api/design.ts`, `api/ai.ts`, etc).
- `errors.ts` — Domain error types and codes.

**Deliverable in `libs/types/openapi/`:**
- One OpenAPI 3 YAML file per microservice describing every endpoint, derived from the DTOs above using a tool like `tsoa` or `nestjs/swagger` generation in reverse.

**Exit criteria:**
- Every PRD-mentioned data shape has a TypeScript type.
- `libs/types` exports a clean public surface; downstream apps import via `@smoxlab/types`.
- An ADR (Architecture Decision Record) committed to `docs/adr/0001-types-canonical.md` documenting that `libs/types` is canonical and services regenerate FROM these types (not the other way around).
- `npm run typecheck` passes with `libs/types` consumed by an intentional throwaway test file in `apps/kiosk` to verify the import path resolves.

### Phase 2 — Database Schema & Migrations (~2–3 weeks)

**Goal:** The Postgres schema that every service will read/write.

**Deliverable in `libs/db/` (new package):**
- Prisma schema (recommend Prisma over TypeORM for solo dev — better TS ergonomics, simpler migrations).
- Tables: `users`, `customers`, `creators`, `operators`, `products`, `product_variants`, `designs`, `design_versions`, `orders`, `order_status_history`, `printer_jobs`, `printers`, `published_designs`, `marketplace_purchases`, `creator_earnings`, `payouts`, `templates`, `prayer_schedules`, `pause_history`, `kiosk_sessions`, `ar_photos` (with TTL), `audit_log`.
- Indexes on `orders(status, created_at)`, `orders(ticket_number)`, `printer_jobs(status, created_at)`, `users(email)`, `published_designs(creator_id, status)`.
- Seed script that inserts: 1 demo store, 5 demo products with variants, 5 demo templates, 1 demo operator account.
- Migration #1 baseline.

**Exit criteria:**
- `npx prisma migrate dev` runs cleanly against a fresh Postgres in Docker.
- `npx prisma generate` produces a Prisma client consumed in a throwaway test.
- Seed script populates demo data; querying the demo store returns the expected products.
- All foreign keys, indexes, and constraints documented in `docs/adr/0002-database-schema.md`.

### Phase 3 — Docker Compose Infra & Service Skeletons (~3–4 weeks)

**Goal:** All 8 NestJS services and 1 gateway boot in Docker Compose, expose `/health`, connect to Postgres + Redis.

**Deliverable:**
- **Write `docs/adr/0001-service-split.md`** finalizing the exact service list before any service implementation begins. **Target: 6 services** (down from the SAD's 8). Consider merging `user-service` into `auth-service` (the auth/user boundary is artificial when there's no separate identity provider) and folding product/catalog endpoints into `order-service` (products are read mostly in the order flow; a separate service adds gateway hops without independent scale needs). Document the rationale for every split-or-merge decision in the ADR. The final list lands here before the rest of Phase 3 proceeds.
- `docker-compose.yml` at repo root with the finalized services from the ADR above, plus: `postgres`, `redis`, `gateway`. Likely landing on: `gateway`, `auth-service` (auth + user merged), `order-service` (order + product merged), `design-service`, `ai-service`, `admin-service`, `printer-service`, `creator-service` — but the ADR is binding.
- One NestJS app per service in `apps/services/{name}-service/` (or `services/{name}/` — decide and document).
- Each service:
  - Boots cleanly inside its Docker container.
  - Reads config from env vars (DB URL, Redis URL, JWT secret stub, service port).
  - Exposes `GET /health` returning `{ ok: true, service: '<name>', version: '<version>', db: 'ok'|'down', redis: 'ok'|'down' }`.
  - Connects to Postgres via Prisma client.
  - Connects to Redis (ioredis).
  - Logs structured JSON to stdout (Pino).
  - Has a `Dockerfile` with multi-stage build.
- Gateway:
  - NestJS app that proxies `/api/<service>/...` to the corresponding service.
  - JWT validation middleware in place (no issuance yet — that's Phase 4).
  - Rate limiting middleware in place (no aggressive limits yet — just wired).
  - CORS allowlist for `localhost:3000`, `localhost:3001`, `localhost:3002` (the 3 frontends).
- A make/just/npm script `docker compose up` brings everything up.

**Exit criteria:**
- `docker compose up` brings all 11 containers (10 services + postgres + redis) to a healthy state in < 60 seconds.
- `curl http://localhost:8080/api/order/health` returns `{ ok: true, service: 'order-service', db: 'ok', redis: 'ok' }`.
- Killing the Postgres container and restarting it: services reconnect within 30 seconds without manual intervention.
- An `ARCHITECTURE.md` documenting the port map and the proxy routing.

### Phase 4 — Auth Service Implementation (~4–6 weeks)

**Goal:** Real auth flows working end-to-end (backend-only first — no UI).

**Deliverable:**
- `auth-service`:
  - `POST /auth/signup` — email + password signup; sends verification email (logged to console in dev).
  - `POST /auth/login` — returns JWT access + refresh tokens.
  - `POST /auth/refresh` — token rotation.
  - `POST /auth/logout` — revokes refresh token (Redis blocklist).
  - `POST /auth/forgot-password`, `POST /auth/reset-password` — token-based reset.
  - Password hashing: Argon2id.
  - Social auth stubs: Google, Apple OAuth callback handlers (no real OAuth keys yet — just the endpoints and DB shape).
  - Role-based access: `customer | creator | operator | admin`.
- `user-service`:
  - `GET /users/me` — returns current user profile.
  - `PATCH /users/me` — update profile.
  - `GET /users/:id` (admin-only).
- Integration tests: signup → login → /users/me → logout → /users/me (401).

**Exit criteria:**
- Postman / Bruno collection in `docs/api/auth.bru` walks through the full signup-login-refresh-logout cycle.
- JWT validation middleware in gateway rejects unauthenticated calls to protected routes.
- All `auth-service` endpoints have integration tests against a real Postgres in Docker.
- `docs/adr/0003-auth-strategy.md` documents JWT scheme, refresh rotation policy, social-auth deferral plan.

### Phase 5 — Product, Order, & Printer Service Implementations (~6–8 weeks)

**Goal:** The core order lifecycle, end-to-end, with a mocked printer.

**Deliverable:**
- `product-service` (or rolled into another service — decide and document):
  - `GET /products` — list available products with variants.
  - `GET /products/:id` — detailed product info.
  - Admin endpoints for product CRUD (gated by `operator | admin` role).
- `order-service`:
  - `POST /orders` — create order from a kiosk session (anonymous OK).
  - `GET /orders/:ticketNumber` — public lookup by ticket.
  - `PATCH /orders/:id/status` — operator-only status updates.
  - WebSocket gateway: emits `order:status-changed` events on every status transition.
  - Ticket number generation (collision-free, short, human-readable — e.g. 6-digit base32).
- `printer-service`:
  - **Mocked printer adapter** that simulates a real DTG printer:
    - Accepts a print job via Redis queue (BullMQ).
    - Transitions through `QUEUED → PRINTING (30–90s simulated) → DONE`.
    - 10% random failure rate to exercise error paths.
  - WebSocket gateway: emits `printer:status-changed` and `printer:job-completed` events.
  - Adapter interface designed so a real printer can be slotted in later (Phase 11).
- Integration tests: create order → it flows through the mocked printer → status reaches `READY`.

**Exit criteria:**
- Hitting `POST /api/order` with a valid payload creates an order, queues a print job, and within 30–90 seconds the order status transitions to `READY` (visible via `GET /api/order/:ticket`).
- Subscribing to the WebSocket order-status stream during this flow yields 4–5 events in correct sequence.
- 10% of orders fail (mocked) and the error path is observable in logs and via the admin endpoint.
- The mocked printer adapter implements the same TypeScript interface that a real printer adapter will implement later.

### Phase 6 — Design, AI, and Creator Service Implementations (~6–8 weeks)

**Goal:** All remaining backend services completed with mocked external dependencies.

**Deliverable:**
- `design-service`:
  - `POST /designs` — create/update a design (JSON-serialized Konva state + asset references).
  - `GET /designs/:id`, `GET /users/me/designs`.
  - Design versioning (store full history for an opt-in retention window — PRD says 90 days).
  - Asset upload endpoint: `POST /designs/:id/assets` — accepts PNG/JPG/SVG, stores to a local MinIO container (S3-compatible) in dev, real S3/R2 in prod.
  - Background-removal stub (calls a mocked endpoint returning the input image unchanged for now).
- `ai-service`:
  - `POST /ai/generate` — accepts a prompt, returns 4 image URLs.
  - **Mocked provider:** returns 4 pre-generated stock images from a fixture set. Latency simulated at 2–5 seconds.
  - **Real provider interface** designed so OpenAI / Stable Diffusion can be slotted in via env var.
  - `POST /ai/moderate` — accepts an image, returns `{ safe: true | false, reasons: string[] }`. Mocked to always return `safe: true` for now; real OpenAI Moderation API integration deferred to a later sub-phase.
  - Credit tracking: each generation decrements user credit balance.
- `creator-service`:
  - `POST /creators` — creator onboarding from an existing user account.
  - `POST /creators/me/designs` — publish a design to the marketplace.
  - `GET /marketplace/designs` — public search/filter.
  - `POST /marketplace/purchases` — record a marketplace purchase (single-use unlock).
  - Earnings ledger: every marketplace sale credits the creator at their configured commission rate.
  - Payout request endpoint (no real payment integration — just a row in `payouts` table for ops to process manually for v1).

**Exit criteria:**
- Hitting `/api/ai/generate` returns 4 mocked image URLs within 5 seconds.
- A creator can sign up → publish a design → the design appears in `/api/marketplace/designs`.
- A purchase via `/api/marketplace/purchases` increments the creator's earnings balance.
- Integration tests cover all critical happy paths and the moderation block path (when mocked to fail).

### Phase 7 — Admin Service & Prayer Pause Service (~3–4 weeks)

**Goal:** The remaining cross-cutting backend concerns.

**Deliverable:**
- `admin-service`:
  - Aggregate dashboard endpoints: `GET /admin/orders/today`, `GET /admin/printers/status`, `GET /admin/analytics/daily`.
  - Prayer schedule management: `GET /admin/prayer/schedule`, `PATCH /admin/prayer/config`, `POST /admin/prayer/pause`, `POST /admin/prayer/resume`.
  - Audit log writes for every admin action.
- **Prayer Pause logic** (lives in `admin-service` and is consumed via WebSocket events by all frontends + the printer service):
  - Use the `adhan` library (or equivalent) for prayer time calculation with the Moroccan Ministry of Habous method as default.
  - Cron-style scheduler triggers `prayer:pre-warning` (5 minutes before), `prayer:pause-start`, `prayer:pause-end` events.
  - Printer service subscribes to these events and pauses the queue accordingly (in-flight jobs allowed to complete; new jobs blocked).
  - Cached schedule persisted to Redis with a 7-day TTL for offline resilience.

**Exit criteria:**
- Setting the system clock 5 minutes before a configured prayer time (or using a `forcePrayerEvent` debug endpoint) triggers the full pause flow: pre-warning event → pause-start event → printer queue blocks new jobs → pause-end event → printer queue resumes.
- All events visible in WebSocket subscribers.
- Operator can manually override via `/api/admin/prayer/pause` and `/api/admin/prayer/resume`.
- Pause history is logged with start time, resume time, and orders affected.

## Conductor Parallelization Opportunities

Some phases can run as parallel Conductor workspaces to compress the calendar timeline. These pairings are bounded by Phase 1's contracts being stable — once contracts are locked, independent service work can fan out.

**Parallelizable phase pairs:**

- **Phase 4 (auth-service) ‖ Phase 5 (order-service + printer-service)** — auth and order/printer are independent backend domains. Once Phase 1 contracts are stable and Phase 3 infra is up, these can run as two parallel workspaces with periodic sync at the gateway layer. Merge weekly to catch contract drift early.
- **Phase 10 (queue-display app) ‖ Phase 12 (AI generation UI in kiosk)** — different frontends, different services, no shared state during build. Run as parallel workspaces once Phase 9 (kiosk core flow) and Phase 8 (frontend foundations) are complete.
- **Phase 6 internal split: design-service ‖ ai-service ‖ creator-service** — three independent backend domains that mostly don't touch each other. Three parallel workspaces possible, with creator-service blocked until design-service's publish API stabilizes (week 2–3 of the phase).
- **Phase 13 (web customer flow) ‖ Phase 14 (marketplace + creator dashboard)** — same frontend codebase, but the marketplace domain is independent enough to split into parallel workspaces if merge discipline is tight.
- **Phase 15 (admin dashboard) ‖ Phase 16 (offline PWA)** — different apps, different concerns. Parallel workspaces work cleanly.

**Intentionally serial:**

- **Phase 11 (AR try-on)** is the highest-risk single slice in the entire build: ≥24 fps target on browser hardware, MediaPipe + WebGL + Three.js + sync-back to design canvas. Trying to parallelize anything against AR pulls focus from the single phase most likely to derail the project. Run it serial, full attention, 12–16 weeks of solo focus.
- **Phases 0, 1, 2, 3 (foundations)** are upstream of everything and must complete sequentially. Parallelizing them produces contract drift before contracts exist.
- **Phase 17 (hardening)** sweeps across all services and apps — parallel work risks merge chaos because every codebase is touched.

**Throughput realism:** Parallel doesn't free calendar time linearly — solo dev attention is the bottleneck. Two parallel workspaces typically yield ~1.3–1.5x throughput, not 2x. Use parallelization mostly to keep cognitive load varied (a backend day vs a frontend day) rather than to compress the calendar arbitrarily. The biggest gains come from running tedious mechanical work (Phase 1 type definitions, Phase 4 auth boilerplate) in one workspace while exploratory work (Phase 11 AR research spikes) runs in another.
